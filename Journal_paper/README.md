# How Are HPC, Edge, and Serverless Architectures Built in the Cloud Continuum?

## An Analysis of Real Cloud Architectures in AWS

Reproducibility package for the journal paper *“How Are HPC, Edge, and Serverless Architectures Built in the Cloud Continuum? An Analysis of Real Cloud Architectures in AWS”* by Steven Santillán and [Cristina L. Abad](https://sites.google.com/fiec.espol.edu.ec/cv-cabad/english) from [ESPOL University](https://www.espol.edu.ec/en).

This package reproduces the analyses of **396 real-world AWS architecture graphs** from the Cloudscape dataset. The study investigates how High-Performance Computing (HPC), Edge, and Serverless architectures are combined in practice, and further examines Strictly Edge, Function-as-a-Service (FaaS), Serverless Compute, and Event-Driven subcategories.

The package is organized as four focused Jupyter notebooks. They share processed data artifacts rather than relying on variables left in the memory of another notebook. This makes each analytical component easier to inspect and reproduce independently.

## Data Artifacts

### `architectures_with_services.pkl`

**Role:** Intermediate, Python-native dataset.

**Contents:** 396 rows, one per Cloudscape architecture graph, with:

- `architecture`: Cloudscape GraphML identifier, derived from the filename.
- `services`: list of AWS services and actor/service labels extracted from graph nodes.

The file is generated by the first code cell of `01_analyze_architecture_characteristics.ipynb` after cloning the public Cloudscape repository and reading all GraphML files.

The `.pkl` format intentionally preserves `services` as genuine Python lists. This avoids ambiguity that can occur when list-valued columns are stored in CSV form and is useful for analyses that repeatedly expand, count, normalize, or compare services.

**Relationship to the workflow:** This is an intermediate artifact. It is generated before classification and is also retained in the package as a convenient checkpoint for inspecting the service-extraction stage.

---

### `classified_architectures_with_metadata.csv`

**Role:** Main processed dataset and primary input for the graph-metric, gSpan, and temporal notebooks.

**Contents:** 396 rows and 28 columns. It combines the extracted services, architecture metadata from the GraphML files, internal labels, and analytical category flags.

The principal column groups are:

| Column group | Main fields | Description |
|---|---|---|
| Architecture identity | `architecture`, `name`, `description`, `category`, `link` | Cloudscape identifier and GraphML metadata, including the associated source/video link. |
| Base classifications | `tipo_arquitectura`, `tipo_arquitectura_no_cloudfront` | Curated internal labels based on the Edge and HPC discriminator rules, with an alternative Edge classification that excludes CloudFront. |
| Service information | `services` | Full list of services associated with the architecture. The values are stored as Python-list text in the CSV and are parsed by the notebooks when list operations are needed. |
| Serverless detection | `serverless_found`, `num_serverless`, `has_serverless` | General Serverless service detection and the corresponding count/flag. |
| Serverless subcategories | `serverless_faas_found`, `serverless_compute_found`, `serverless_event_driven_found`, related count and Boolean fields | Detected FaaS, Serverless Compute, and Event-Driven services and membership indicators. |
| Analytical membership flags | `belongs_edge`, `belongs_hpc`, `belongs_serverless`, `belongs_none_analytical`, `belongs_edge_no_cloudfront`, and subcategory flags | Boolean indicators used to construct the groups throughout the package. |

The data model permits overlapping categories:

- **Edge** includes both `Edge` and `Edge+HPC` internal labels.
- **HPC** includes both `HPC` and `Edge+HPC` internal labels.
- **Serverless** is an independent, overlapping category based on the presence of one or more Serverless services.
- **None** contains architectures that are not in the analytical Edge, HPC, or Serverless groups.
- **Strictly Edge** refers to the Edge scenario in which CloudFront is not treated as an Edge discriminator.
- **FaaS**, **Serverless Compute**, and **Event-Driven** are overlapping Serverless subcategories.

This CSV is generated in `01_analyze_architecture_characteristics.ipynb` after applying the classification rules, curated manual overrides, Serverless detection, metadata extraction, and integrity checks.

---

### `architectures_with_year.csv`

**Role:** Derived temporal-analysis dataset.

**Contents:** 396 rows and 29 columns. It preserves every field from `classified_architectures_with_metadata.csv` and adds:

- `year_video`: publication year retrieved from the YouTube video associated with the architecture link.

The current packaged file contains publication-year information for 395 architectures, with years ranging from **2019 to 2023**. Missing values are retained when a year cannot be retrieved.

This file is generated by `04_analyze_temporal_trends.ipynb` using the YouTube Data API v3. It is included as a precomputed artifact so that the temporal figures and tables can be reproduced without repeatedly querying the API.

## Reproducibility Notebooks

### 1. [Analyze Architecture Characteristics](./01_analyze_architecture_characteristics.ipynb)

**Purpose:** Main classification and descriptive-analysis notebook.

This notebook starts from the original Cloudscape GraphML graphs. It clones the public Cloudscape repository, extracts node-level services, classifies architectures, merges GraphML metadata, and performs the descriptive analyses used throughout the paper.

#### Main processing stages

1. **Service extraction**
   - Reads the Cloudscape GraphML files.
   - Extracts node attributes labelled as services.
   - Produces `architectures_with_services.pkl`.

2. **Architecture classification**
   - Defines Edge, HPC, and Serverless discriminator-service lists.
   - Builds the internal labels `Edge`, `HPC`, `Edge+HPC`, and `None`.
   - Applies curated manual overrides.
   - Builds the alternative no-CloudFront Edge classification.
   - Identifies Serverless, FaaS, Serverless Compute, and Event-Driven services.
   - Produces `classified_architectures_with_metadata.csv`.
   - Creates `labeled_architectures.pkl` locally for downstream cells in the same runtime.

3. **Descriptive analyses**
   - Category prevalence and category intersections.
   - Serverless composition and sensitivity scenarios excluding Lambda or S3.
   - Frequently used services and category-specific discriminators.
   - Service co-occurrence analysis.
   - Strictly Edge analysis.
   - Number-of-services distributions and box plots.
   - Storage-service analysis.
   - Machine-learning-service analysis.
   - Functional-goal/category analysis.
   - Workflow-count analysis.
   - Industry analysis.

#### Important inputs

- Public [Cloudscape repository](https://github.com/WiscADSL/Cloudscape), cloned by the notebook.
- A translated industry dictionary retrieved from the repository's workshop package for the industry-analysis section.

#### Main outputs

The notebook creates the two package datasets described above and additionally generates analysis-specific CSV, EPS, and PNG outputs, including:

- Category and service-frequency tables.
- Service co-occurrence tables.
- Service-count descriptive statistics and box plots.
- Storage and machine-learning usage summaries.
- Functional-goal tables and visualizations.
- Workflow summaries and workflow box plots.
- Industry counts, percentages, and visualizations.

**Execution note:** Run this notebook from its first code cell when starting a fresh Colab runtime. Several later cells load `labeled_architectures.pkl`, which is created by the classification block earlier in the same notebook.

### 2. [Analyze Graph Metrics](./02_analyze_graph_metrics.ipynb)

**Purpose:** Structural analysis of the Cloudscape GraphML architecture graphs.

The notebook independently clones Cloudscape and downloads `classified_architectures_with_metadata.csv`. It does not depend on the in-memory state of Notebook 1.

#### Graph-level metrics

For every GraphML architecture graph, the notebook computes:

- Number of nodes and edges.
- Graph density.
- Average degree, average in-degree, and average out-degree.
- Maximum path length.
  - Uses a longest-path calculation for directed acyclic graphs.
  - Uses condensation into a DAG when directed cycles are present.
  - Uses component diameters for undirected graphs.
- Cycle indicator and path-computation method.
- Branching nodes, additional branches, and maximum branching factor.
- Number of source and sink nodes.
- Weakly and strongly connected component counts.
- Number of observed service types.
- Shannon entropy and normalized entropy of service-type distribution.

#### Category-level analyses

The graph-level metrics are joined with the category flags in `classified_architectures_with_metadata.csv`. The notebook produces category-level summaries for HPC, Edge, Serverless, None, Strictly Edge, FaaS, Serverless Compute, and Event-Driven architectures.

#### Main outputs

- `graph_metrics.csv`: graph-level metrics for all Cloudscape graphs.
- Category membership and category-count CSV files.
- Long-format, summary, and mean graph-metric tables by analytical category.
- One graph-metrics CSV per selected category.
- `graph_metrics_by_category_table_numeric.csv`: numeric values used to build the manuscript table.
- `graph_metrics_by_category_table.csv`: display-formatted version of the graph-metric table.

### 3. [Mine Frequent Graph Patterns](./03_mine_frequent_graph_patterns.ipynb)

**Purpose:** Frequent subgraph mining and bootstrap validation using gSpan.

This notebook independently clones Cloudscape, downloads `classified_architectures_with_metadata.csv`, and converts the GraphML graphs into gSpan input files.

#### Main processing stages

1. **Global GraphML-to-gSpan conversion**
   - Creates integer catalogs for node/service labels and edge types.
   - Exports a global gSpan dataset and graph-ID mapping files.

2. **Category-specific graph preparation**
   - Uses the analytical category flags to select the HPC, Edge, Serverless, None, Strictly Edge, and FaaS graph sets.
   - Validates that the architectures in the metadata CSV match available GraphML files.
   - Produces a separate normalized gSpan input file for each analytical category.

3. **Label normalization**
   - Consolidates variants of user/actor labels into generalized categories such as `UserCompany` and `UserConsumer`.
   - Preserves service and edge-type catalogs that allow numeric gSpan output to be interpreted as readable patterns.

4. **Frequent-pattern mining**
   - Mines patterns with 2 to 4 nodes.
   - Uses normalized minimum-support ratios of 10%, 6%, and 3% within each category.
   - Exports raw gSpan outputs and parsed, human-readable pattern CSV files.
   - Produces the consolidated file `all_patterns_normalized_by_category_v2.csv`.

5. **Bootstrap validation**
   - Resamples eligible category graphs with a fixed random seed.
   - Measures pattern recovery, frequency stability, support summaries, confidence intervals, category selection, and execution errors.
   - The notebook contains a short test bootstrap configuration and a final configuration with 1,000 iterations. Run only the intended bootstrap cell for the analysis you wish to perform.

#### Main output locations

Global and category-specific gSpan artifacts are written below the cloned Cloudscape directory, including:

```text
/content/Cloudscape/
├── cloudscape_gspan.data
├── gspan_node_labels.json
├── gspan_edge_labels.json
├── gspan_graph_names.json
│
└── gspan_normalized_by_category/
    ├── gspan_input_files/
    ├── raw_gspan_outputs/
    ├── parsed_results/
    │   └── all_patterns_normalized_by_category_v2.csv
    └── bootstrap_validation/
        └── results/
            ├── bootstrap_pattern_stability.csv
            ├── bootstrap_group_summary.csv
            ├── bootstrap_iteration_metrics.csv
            ├── bootstrap_category_selection.csv
            ├── bootstrap_execution_errors.csv
            └── bootstrap_baseline_summary.csv
```

### 4. [Analyze Temporal Trends](./04_analyze_temporal_trends.ipynb)

**Purpose:** Temporal analysis of the architecture collection based on YouTube publication years.

The notebook first downloads `classified_architectures_with_metadata.csv`, then retrieves publication years from the YouTube Data API v3 by extracting video identifiers from the `link` field.

#### Main processing stages

1. **Video-year retrieval**
   - Supports standard YouTube, shortened, embedded, and Shorts URL formats.
   - Retrieves the publication year through the YouTube Data API v3.
   - Adds the resulting `year_video` field.
   - Produces `architectures_with_year.csv`.

2. **Temporal aggregation and plotting**
   - Builds annual counts for HPC, Edge, Serverless, None, Strictly Edge, and FaaS.
   - Produces both absolute counts and percentages relative to the architectures observed in each year.
   - Exports publication-ready EPS figures and supporting CSV tables.

#### Main outputs

- `architectures_with_year.csv`
- `absolute_evolution_year_strictly_edge_faas.eps`
- `percentage_evolution_year_strictly_edge_faas.eps`
- `absolute_evolution_year_strictly_edge_faas.csv`
- `percentage_evolution_year_strictly_edge_faas.csv`
- `total_architectures_year.csv`

#### YouTube API requirement

To rerun the video-year retrieval cell in Google Colab, store a YouTube Data API v3 key as a Colab Secret with the exact name:

```text
Api-YT
```

The key must not be written directly in the notebook or committed to the repository.

**API-free reproduction option:** The included `architectures_with_year.csv` is the precomputed output of the retrieval step. To reproduce only the temporal plots and tables without a YouTube API key, make this file available at:

```text
/content/architectures_with_year.csv
```

and run the trend-analysis code cell that reads this file.

## Recommended Execution Paths

### Full regeneration from the original Cloudscape graphs

1. Run `01_analyze_architecture_characteristics.ipynb` from top to bottom.
2. Run `02_analyze_graph_metrics.ipynb`.
3. Run `03_mine_frequent_graph_patterns.ipynb`.
4. Run `04_analyze_temporal_trends.ipynb`, supplying the `Api-YT` Colab Secret to rerun the year-retrieval step.

### Reproduce a specific analytical component

- **Architecture characteristics:** Run Notebook 1 from the beginning because it generates its local `labeled_architectures.pkl` checkpoint.
- **Graph metrics:** Run Notebook 2 independently. It obtains Cloudscape and the classification CSV itself.
- **Frequent graph patterns:** Run Notebook 3 independently. It obtains Cloudscape and the classification CSV itself.
- **Temporal figures without API access:** Use the included `architectures_with_year.csv` and run only the plotting portion of Notebook 4.
